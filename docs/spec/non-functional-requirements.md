# 非機能要件 (Non-Functional Requirements)

本書は「OpenCV Playground」の非機能要件を定義する。OpenCV Playground は、OpenCV.js (WebAssembly) をブラウザ内で実行し、ユーザーがアップロードした画像に対して OpenCV の画像処理機能を試せる、サーバーレスな React + TypeScript SPA である。バックエンドを持たず、画像はブラウザの外に一切送信されない。静的ホスティング (GitHub Pages / Netlify / Cloudflare Pages) で配信する。20 カテゴリ・91 デモを提供し、MVP では「1 枚のアップロード済み静止画 + Before/After 表示 + パラメータ操作 + コード例」に焦点を当てる。

機能要件 (各デモの仕様) は別書に定義し、本書はアーキテクチャ・性能・メモリ・互換性・法務 (ライセンス) など、横断的・品質的な要件を扱う。

凡例:
- **要件**: その節で満たすべき非機能要件 (What)。
- **方針 / 実装指針**: 実現のための具体的なアプローチ (How)。
- 現時点で確証が持てない事項は、断定せず明示的に **要調査** と記す。

---

## 1. アーキテクチャ方針 (SPA / クライアントサイド完結)

### 要件

- アプリケーションは単一ページアプリケーション (SPA) として実装し、すべての画像処理ロジックをクライアントサイド (ブラウザ) で完結させる。
- サーバーサイドでの動的処理 (画像処理、API、DB アクセス) を一切持たない。実行時に必要なのは静的アセットの配信のみとする。
- React + TypeScript を採用し、型安全性とコンポーネント単位の保守性を確保する。

### 方針 / 実装指針

- **レイヤ構成 (論理):**
  - **UI 層**: React コンポーネント (画像表示、パラメータ UI、コード表示、カテゴリ/デモ一覧)。
  - **ドメイン層**: 各デモの処理定義 (入力・パラメータ仕様・OpenCV.js 呼び出し・出力)。デモは「ID・カテゴリ・名称 (日/英)・パラメータスキーマ・処理関数・コード例」を持つ純粋なデータ + 関数として記述し、UI から疎結合に保つ。
  - **OpenCV ラッパ層**: OpenCV.js の初期化・Mat 生成/破棄・実行を抽象化し、UI 層が `cv` グローバルを直接触らないようにする。
- **状態管理**: MVP の状態は限定的 (選択中デモ、入力画像、パラメータ値、処理結果、ロード状態) のため、まずは React の `useState` / `useReducer` + Context で構成する。グローバルな状態ライブラリ (Zustand 等) の導入は規模拡大時に再検討する。
- **ルーティング**: クライアントサイドルーティング (React Router 等) を用い、`/demo/:category/:id` のような URL でデモを直接共有できるようにする。SPA fallback の扱いは第 2 節を参照。
- **ビルド**: Vite を推奨 (高速な開発サーバ、ESM ベース、動的 import / コード分割が容易、`base` パス設定が単純で GitHub Pages のサブパス配信に対応しやすい)。

## 2. サーバーレス・静的ホスティング (ビルド成果物のみ配信)

### 要件

- 配信物はビルド成果物 (HTML / JS / CSS / WASM / 画像等の静的アセット) のみとする。サーバープロセスや実行環境を前提としない。
- 主要ホスティング候補 (GitHub Pages / Netlify / Cloudflare Pages) のいずれでも動作すること。
- CDN によるエッジ配信でグローバルに低レイテンシで配信できること。
- SPA であるため、任意のルートへの直接アクセス・リロードでも正しく `index.html` が返ること (SPA fallback routing)。

### 方針 / 実装指針

- **SPA fallback routing**: 404 を `index.html` にフォールバックさせ、クライアントルータに解決を委ねる。
  - GitHub Pages: `404.html` に `index.html` 相当を配置する手法、または HashRouter の採用を検討。サブパス配信時は Vite の `base` を正しく設定する。
  - Netlify: `_redirects` に `/* /index.html 200` を記述。
  - Cloudflare Pages: SPA モード (未マッチを `index.html` にフォールバック) を設定。
- **CDN / キャッシュ戦略**:
  - ハッシュ付きアセット (`app.[hash].js` 等) は `Cache-Control: public, max-age=31536000, immutable` で長期キャッシュ。
  - `index.html` は短期キャッシュまたは `no-cache` とし、デプロイ後の更新が反映されるようにする。
  - `opencv.js` / `.wasm` は大きい (第 4 節) ため、ファイル名にバージョン/ハッシュを含め、長期キャッシュ + 内容変更時のみ再取得とする。
- **設定の宣言的管理**: ホスティング差異を吸収するため、リダイレクト/ヘッダ設定はリポジトリ内に宣言的に持つ (例: `public/_redirects`, `public/_headers`, ワークフロー設定)。
- **COOP/COEP ヘッダ**: WASM threads (SharedArrayBuffer) を有効化する場合、特定の HTTP レスポンスヘッダが必要となる。静的ホスティングでこれらを付与できるかはホスティングごとに差があり **要調査** (第 4 節参照)。

## 3. ブラウザ内処理・プライバシー

### 要件

- ユーザーがアップロード・撮影した画像、および処理結果を、いかなるサーバーにも送信しない。すべての処理はブラウザのメモリ内で完結する。
- 「画像が端末から出ない」ことを本アプリの明確な価値 (売り) として、UI 上およびドキュメントで明示する。
- 計測 / 解析を導入する場合も、画像内容やそれに由来する情報を外部に送らないこと。

### 方針 / 実装指針

- **送信ゼロの担保**:
  - 画像の入力は `<input type="file">` / ドラッグ&ドロップ / `getUserMedia` (カメラ) のいずれも、取得後は `FileReader` / `createImageBitmap` / `canvas` を介してメモリ上で扱い、`fetch` / `XMLHttpRequest` / WebSocket 等での外部送信を行わない。
  - ビルド時に、画像データを外部送信し得るコードが混入していないことをレビュー観点に含める。
- **プライバシー表示**: トップおよびアップロード UI 付近に「画像はブラウザ内だけで処理され、サーバーに送信されません」と明記する。プライバシーポリシーページ (静的) を用意する。
- **計測 / 解析の扱い**:
  - 導入する場合はプライバシー配慮型 (例: Cookie レス、個人特定を行わないページビュー計測) を優先し、収集項目を最小化する。
  - 画像のピクセルデータ・ファイル名・EXIF 等は計測対象に含めない。
  - エラーレポート (任意導入時) も、画像本体やそれを復元できる情報を送らない設定とする。送信前にペイロードを精査する。
- **EXIF / メタデータ**: 表示・処理にあたり EXIF の Orientation は考慮する (第 8 節)。位置情報など機微なメタデータをログ/計測へ流出させない。

## 4. OpenCV.js / WASM の読み込み方針

### 要件

- `opencv.js` (WASM 含む) はおおよそ 8〜10MB と大きく、初回ロードがアプリのボトルネックになる。これを前提に読み込み戦略を定義する。
- OpenCV ランタイムの初期化完了を確実に検知し、初期化前に `cv` の関数を呼ばないこと。
- 初回ロードのユーザー体験 (UX) を許容範囲に保つこと。
- 2 回目以降の訪問ではキャッシュにより高速化されること。

### 方針 / 実装指針

- **遅延ロード / 動的 import**:
  - `opencv.js` をアプリ初期バンドルに含めず、ユーザーがデモを実行する直前 (またはデモ画面表示時) に動的に読み込む。トップ/一覧の表示は OpenCV ロードを待たせない。
  - 動的 `import()` でラッパモジュールを分離し、コード分割する。`opencv.js` 自体はスクリプトタグ/Worker からの `importScripts` 等、配布形態に応じた方法で読み込む。
- **初期化完了の検知**:
  - OpenCV.js は初期化完了を `Module.onRuntimeInitialized` (または `cv['onRuntimeInitialized']`) で通知する。これを Promise でラップし、`await ready()` 後にのみ処理を実行する。
  - ビルドによっては `cv` が Promise を返す形態 (`cv = await cv()`) のものもあるため、採用するビルドの初期化 API を確認して合わせる。**要調査** (採用ビルドの公式 API)。

  ```ts
  // OpenCV.js ロード + 初期化完了を Promise で待つ例 (スクリプト配置の場合)
  let cvReady: Promise<void> | null = null;

  export function loadOpenCV(scriptUrl: string): Promise<void> {
    if (cvReady) return cvReady;
    cvReady = new Promise<void>((resolve, reject) => {
      const w = window as any;
      // 初期化完了コールバックを先に登録してから読み込む
      w.Module = {
        onRuntimeInitialized: () => resolve(),
        // .wasm の取得先を明示したい場合は locateFile を指定
        // locateFile: (path: string) => `/opencv/${path}`,
      };
      const s = document.createElement("script");
      s.src = scriptUrl;
      s.async = true;
      s.onerror = () => reject(new Error("Failed to load opencv.js"));
      document.head.appendChild(s);
    });
    return cvReady;
  }
  ```

- **初回ロード UX**:
  - 明示的なロード状態 (スピナー + 「OpenCV を読み込み中… (約 8〜10MB)」のような説明) を表示し、進捗が分かるようにする。可能なら fetch のストリームでダウンロード進捗を表示する (バイト数ベース)。
  - ロード中も UI 操作 (パラメータ確認、説明閲覧) を可能にし、実行ボタンのみ無効化する。
  - 失敗時はリトライ手段を提供する (第 12 節)。
- **キャッシュ**:
  - HTTP キャッシュ (長期 + immutable) に加え、必要に応じて Service Worker によるプリキャッシュを検討。ただし大容量 WASM をキャッシュする場合のストレージ消費・更新戦略に注意。Service Worker 導入は MVP では任意とし、将来の高速化策とする。
- **CDN vs self-host**:

  | 方式 | 利点 | 欠点 / 注意 |
  | --- | --- | --- |
  | 公開 CDN 配信版 | 自前で巨大ファイルを持たなくて済む / 共有キャッシュの可能性 | バージョン固定・可用性・ライセンス表記・将来の URL 変更リスク。COOP/COEP やバージョン整合の制御が効きにくい |
  | self-host (同一オリジン) | バージョン固定・可用性を自分で管理 / COOP/COEP・キャッシュヘッダを完全制御 / オフライン化しやすい | リポジトリ/配信に 8〜10MB を含める。CDN 配信は前段の静的ホスティング CDN に依存 |

  - 本プロジェクトはバージョン固定とヘッダ制御の容易さを重視し、**self-host を基本方針**とする (静的ホスティングの CDN 経由で配信)。
- **SIMD / threads (高速化)**:
  - WASM SIMD 対応ビルドを使えるか、threads (pthread) 版を使えるかで処理性能が大きく変わる。
  - threads 版は `SharedArrayBuffer` を必要とし、そのためにレスポンスヘッダ `Cross-Origin-Opener-Policy: same-origin` および `Cross-Origin-Embedder-Policy: require-corp` (COOP/COEP) が必須となる。これは外部リソース (画像 CDN 等) の読み込み制約 (CORP/CORS) を伴うため影響が広い。
  - 公式配布の OpenCV.js が SIMD / threads 有効ビルドとして入手可能か、また採用ホスティングで COOP/COEP を付与できるかは **要調査**。MVP では「単一スレッド WASM (+ SIMD があれば利用)」を前提とし、threads 化は将来の最適化として位置づける。

## 5. パフォーマンス

### 要件

- 一般的なデモの 1 回の処理は、対象解像度の入力に対して体感即時 (おおむね 1 秒未満) を目標とする。重い処理 (一部の特徴量・セグメンテーション等) はこの限りでないが、その場合も処理中であることを明示する。
- 処理中に UI が完全にフリーズしないこと (操作不能・無反応にならない)。
- パラメータのスライダー操作などで過剰に処理が走らないこと。

### 方針 / 実装指針

- **処理時間目標 (目安)**:

  | 区分 | 例 | 目標 |
  | --- | --- | --- |
  | 軽量 | しきい値処理、色変換、リサイズ、ぼかし | < 100ms |
  | 中量 | Canny、モルフォロジー、ヒストグラム均一化 | < 300ms |
  | 重量 | 特徴点検出/マッチング、輪郭+描画、一部フィルタ | < 1s (越える場合は進捗/中断 UI) |

  数値は MVP の上限解像度 (下記) を前提とした目安であり、計測に基づき調整する。
- **UI スレッドのブロック回避**:
  - 重い処理は Web Worker への移譲を検討する (第 7 節)。MVP で同期実行する場合も、実行直前にローディング表示へ切り替え、処理後に結果反映する (最低限、開始を 1 フレーム遅延させて描画を確定させる)。
- **デバウンス / スロットル**:
  - スライダー等の連続入力はデバウンス (例: 入力停止後 100〜200ms) してから処理を実行する。ドラッグ中はプレビューを間引く、または確定時のみ処理するなど、入力 UX に応じて選択する。
  - 直前の処理が完了する前に新しい入力が来た場合は、古い処理結果を破棄 (キャンセル/上書き) し、最新入力のみ反映する。
- **入力画像の上限解像度**: 処理コストとメモリは画素数にほぼ比例する。アップロード時に長辺を上限まで自動縮小する (第 8 節)。MVP では長辺 1280〜1920px を目安とし、計測で決定する。
- **再計算の最小化**: 入力画像から派生する不変な中間結果 (グレースケール変換等) はキャッシュし、パラメータ変更時に再生成しない。

## 6. メモリ管理と `cv.Mat.delete()` の扱い

### 要件

- OpenCV.js は Emscripten ヒープ上のメモリを **手動管理**する。生成した `cv.Mat` / `cv.MatVector` / その他 `cv` オブジェクトは、不要になった時点で必ず `delete()` する。
- 例外発生時でも確実に解放されること (リークしないこと)。
- 長時間利用・連続実行でヒープが増加し続けないこと。

### 方針 / 実装指針

- **基本原則**:
  - `new cv.Mat()`, `cv.matFromImageData()`, `cv.imread()` 等で得たオブジェクトはすべて呼び出し側の責任で `delete()` する。GC では解放されない。
  - 関数内で生成した一時オブジェクトはその関数内で破棄する。戻り値として外へ渡す Mat は、受け取り側が破棄責任を負うことを規約として明示する。
- **try/finally パターン (基本形)**:

  ```ts
  function toGray(src: cv.Mat): cv.Mat {
    const dst = new cv.Mat();
    try {
      cv.cvtColor(src, dst, cv.COLOR_RGBA2GRAY);
      return dst; // 成功時は呼び出し側に破棄責任を移譲
    } catch (e) {
      dst.delete(); // 失敗時はここで解放してからリスロー
      throw e;
    }
  }

  // 一時 Mat を内部だけで使う場合
  function meanBrightness(src: cv.Mat): number {
    const gray = new cv.Mat();
    try {
      cv.cvtColor(src, gray, cv.COLOR_RGBA2GRAY);
      return cv.mean(gray)[0];
    } finally {
      gray.delete(); // 例外有無に関わらず必ず解放
    }
  }
  ```

- **スコープ単位の自動解放ラッパ (推奨)**:
  - delete 漏れを構造的に防ぐため、スコープ内で生成した cv オブジェクトを登録し、スコープ終了時に一括解放するヘルパを用意する。各デモの処理関数はこのスコープ内で記述する規約とする。

  ```ts
  interface Deletable { delete(): void; }

  /** scope 内で track() した cv オブジェクトを終了時に一括 delete する */
  export function withCvScope<T>(fn: (track: <D extends Deletable>(o: D) => D) => T): T {
    const owned: Deletable[] = [];
    const track = <D extends Deletable>(o: D): D => {
      owned.push(o);
      return o;
    };
    try {
      return fn(track);
    } finally {
      // 後入れ先出しで解放 (依存関係を考慮)
      for (let i = owned.length - 1; i >= 0; i--) {
        try { owned[i].delete(); } catch { /* 二重解放等は無視 */ }
      }
    }
  }

  // 利用例: 最終的に画面へ返す結果だけ track せず別管理する
  function blurDemo(src: cv.Mat, ksize: number): cv.Mat {
    return withCvScope((track) => {
      const tmp = track(new cv.Mat());
      cv.GaussianBlur(src, tmp, new cv.Size(ksize, ksize), 0);
      // 結果は呼び出し側へ渡すためコピーして返す (tmp はスコープ終了で解放)
      return tmp.clone();
    });
  }
  ```

  - `cv.Size` / `cv.Scalar` 等の値オブジェクトも実装によっては解放対象となるため、採用ビルドでの扱いを確認する。**要調査**。
- **リーク検出の運用**:
  - 開発時に、処理前後でヒープ使用量や Emscripten の確保サイズの増減を確認できる簡易計測を用意し、連続実行でメモリが単調増加しないことをチェックする。
  - デモ単位の単体テストで「実行 → 結果取得 → 解放後に増加なし」を確認することを目標とする (検出手段の確立は **要調査**)。
- **ライフサイクル連動**: 入力画像の差し替え・デモ切り替え・コンポーネント unmount 時に、保持している Mat 群を確実に破棄する (React の cleanup で解放)。

## 7. Web Worker 利用方針

### 要件

- 重い画像処理は UI スレッドをブロックしないよう、Web Worker での実行を検討する。
- Worker 利用の有無に関わらず、メモリ管理 (第 6 節) の原則は維持する。

### 方針 / 実装指針

- **メリット**:
  - メインスレッドをブロックせず、処理中も UI が応答可能。
  - 重い処理のキャンセル (Worker への新規依頼で古い処理を打ち切る/Worker を作り直す) がしやすい。
- **制約 / 注意**:
  - OpenCV.js を Worker 内で動かす場合、`importScripts()` での読み込みと、Worker コンテキストでの `onRuntimeInitialized` 検知が必要。DOM や `Image`/`canvas` の一部 API は Worker から使えないため、入力は `ImageData` / `ImageBitmap` (transferable) / 生の `ArrayBuffer` で受け渡す。
  - メイン⇔Worker 間のデータ転送はコピーコストがある。`ArrayBuffer` の transfer や `ImageBitmap` の transfer を用いてコピーを避ける。
  - Mat はスレッド間で共有できない。Worker 側で生成・処理し、結果を転送可能な形式で返す。
  - threads 版 OpenCV.js を使う場合は SharedArrayBuffer/COOP/COEP の制約が加わる (第 4 節)。
  - Worker 用に `opencv.js` を別途ロードするため、メイン側と二重ロードにならないよう構成する (基本は Worker 側のみでロード)。
- **MVP での方針**:
  - MVP では Worker 利用は **任意**とし、まずはメインスレッド実行 + ローディング表示で許容できるか計測する。
  - 重量デモで体感が悪い、または将来 91 デモ全体の品質を担保する段階で、**Worker 化を必須**へ格上げする。アーキテクチャは最初から「処理関数を Worker へ移しやすい純粋関数」として設計しておく (入力/出力をシリアライズ可能な形に寄せる)。

## 8. 大きな画像のリサイズ方針

### 要件

- アップロードされた画像が大きい場合、処理性能・メモリを守るため、長辺を一定 px 以下へ自動縮小してから処理する。
- 元解像度を保持したい需要 (高解像度での確認・保存) への対応方針を定める。

### 方針 / 実装指針

- **自動縮小**:
  - アップロード時に長辺が上限を超える場合、アスペクト比を保って縮小する。縮小は `createImageBitmap` の `resizeWidth`/`resizeHeight` または canvas 描画で行い、その結果を処理入力 (Mat) とする。
  - EXIF Orientation を考慮して正しい向きに補正してから縮小する。
- **解像度上限の目安**:

  | 用途 | 長辺上限 (目安) | 備考 |
  | --- | --- | --- |
  | デスクトップ標準 | 1920px | 体感とメモリのバランス重視 |
  | モバイル | 1280px (場合により 1024px) | メモリ制約・発熱を考慮 (第 9 節) |
  | 重量デモ | 1024px | 特徴量・反復処理など |

  値は計測に基づき確定する。端末・デモに応じて動的に決めることも検討。
- **元解像度保持の可否**:
  - MVP では「縮小後画像での処理」を基本とし、Before/After も縮小後で表示する。元解像度での処理・保存は、メモリと処理時間のリスクが大きいため MVP の標準とはしない (将来オプションとして検討)。
  - 縮小していることをユーザーに表示する (例: 「表示/処理は最大 1920px に縮小しています」)。
- **メモリ見積り**:
  - OpenCV 内部の Mat は概ね 1 画素あたり 4 byte (RGBA/8UC4) や 1〜3 byte (グレー/カラー) を要する。例: 1920×1080 の RGBA Mat ≒ 1920×1080×4 ≒ 約 8.3MB。処理過程で複数の中間 Mat を保持するため、実使用はこの数倍になり得る。
  - 上限解像度はこの見積りと端末メモリ (特にモバイル) を踏まえて決める。中間 Mat 数を減らす実装を心がける (第 6 節)。

## 9. スマホ・モバイル対応

### 要件

- スマートフォン/タブレットでも主要デモが利用できること。端末性能・メモリ・操作系・発熱・カメラの差異を考慮する。

### 方針 / 実装指針

- **端末性能差**: モバイル CPU は処理が遅い。上限解像度をモバイルで下げ (第 8 節)、重量デモはモバイルで警告/簡略化または実行前の同意を求める。
- **メモリ制約**: モバイルブラウザ (特に iOS Safari) はメモリ上限が厳しく、超過するとタブがクラッシュ/リロードされる。中間 Mat の最小化、上限解像度の引き下げ、不要 Mat の即時解放を徹底する。WASM の 8〜10MB ロードと合わせ、メモリ予算を保守的に見積もる。
- **タッチ操作**: スライダー等のコントロールを十分なタップ領域 (目安 44×44px 以上) とし、Before/After のスワイプ比較やピンチズーム等タッチ前提の操作を検討。ホバー前提の UI を避ける。
- **発熱 / バッテリ**: 連続処理は発熱の原因となる。デバウンスと不要処理の抑制 (第 5 節)、重量処理の頻度制限を行う。
- **カメラ**: `getUserMedia` でのカメラ入力を提供する場合、HTTPS 必須 (第 13 節)、背面/前面カメラ切替、権限拒否時のフォールバック (ファイル選択) を用意する。MVP の主対象はアップロード静止画であり、カメラ入力は将来拡張として扱う。
- **レスポンシブ**: 縦長画面での Before/After 配置 (横並び→縦積み)、コード例の折りたたみ等、画面サイズに応じたレイアウトを提供する。

## 10. アクセシビリティ

### 要件

- 主要機能をキーボードのみで操作でき、スクリーンリーダで意味が伝わること。
- 視覚・運動・前庭感覚などの多様な利用者に配慮する。目安として WCAG 2.1 AA を志向する。

### 方針 / 実装指針

- **WCAG 目安**: 達成基準 AA を目標 (達成度は段階的に高める)。
- **キーボード操作**: すべての操作要素 (デモ選択、ファイル選択、スライダー、実行、ダウンロード) をキーボードで到達・操作可能にする。フォーカス可視化、論理的なタブ順、フォーカストラップの適切な扱いを行う。
- **スクリーンリーダ**: 適切なセマンティック HTML と、必要箇所のみ ARIA を用いる。スライダーは `role`/`aria-valuenow`/`aria-valuetext` 等で現在値を読み上げ可能にする。処理完了・エラーは `aria-live` 領域で通知する。Before/After 画像には意味のある代替テキストを付与する (画像内容は説明困難なため、少なくとも「処理前/処理後の画像」と役割を伝える)。
- **コントラスト**: テキスト/UI のコントラスト比を AA 基準 (通常テキスト 4.5:1、大きいテキスト 3:1) 以上にする。
- **prefers-reduced-motion**: モーション低減設定を尊重し、アニメーション/トランジションを抑制または無効化する。
- **代替手段**: 数値直接入力 (スライダーの代替)、ドラッグ&ドロップに対する通常のファイル選択ボタンの併設など、複数の操作手段を提供する。

## 11. SEO

### 要件

- 公開サイトとして、各デモが検索・共有経由で発見されやすいこと。SPA 特有の SEO 課題に対処する。

### 方針 / 実装指針

- **SPA の SEO 課題**: クライアントレンダリングのみだと、初期 HTML にコンテンツが乏しく、クローラやリンクプレビュー (OGP) が内容を取得できないことがある。
- **プリレンダリング / SSG (静的書き出し)**:
  - ビルド時に各デモ・カテゴリのページを静的 HTML として書き出すプリレンダリングを検討する (デモ数 = 91 + カテゴリ 20 程度で全列挙が現実的)。各 HTML にタイトル・説明・OGP を埋め込み、JS 読み込み後に SPA としてハイドレートする。
  - 採用方法 (Vite のプリレンダリングプラグイン、または SSG 対応フレームワークへの移行) は規模と運用を踏まえて選定する。**要調査**。
  - 注意: OpenCV 処理自体はクライアント実行のため、プリレンダリングするのは「説明・パラメータ仕様・コード例・静的サンプル画像」までで、実処理結果は含めない。
- **メタタグ**: 各ルートで `<title>` / `meta description` / canonical を適切に設定する。SPA でのルート遷移時にもこれらを動的に更新する。
- **デモごとの OGP**: デモごとに OGP (og:title / og:description / og:image) を設定し、SNS 共有時に内容が伝わるようにする。`og:image` はデモを象徴する静的サンプル画像 (Before/After の例) を用意する。OGP は静的 HTML 側に埋め込まれている必要がある (上記プリレンダリングと連動)。
- **sitemap / robots**: `sitemap.xml` と `robots.txt` を生成・配置し、クロールを支援する。
- **構造化データ**: 必要に応じて (アプリ/記事の) 構造化データ付与を検討 (任意)。

## 12. エラーハンドリング

### 要件

- OpenCV.js のロード失敗、未対応 API 呼び出し、処理中の例外、入力不正、ブラウザ非対応など、起こり得る失敗を分類し、ユーザーに分かりやすくフィードバックする。失敗してもアプリ全体が壊れないこと。

### 方針 / 実装指針

- **エラー分類と対応**:

  | 種別 | 例 | ユーザーへの対応 |
  | --- | --- | --- |
  | OpenCV.js ロード失敗 | ネットワーク断、CDN/配信エラー、タイムアウト | 「読み込みに失敗しました」+ 再試行ボタン。回線確認の案内 |
  | 初期化失敗 | `onRuntimeInitialized` 未到達、WASM 非対応 | 非対応の可能性を案内し、対応ブラウザを提示 (第 13 節) |
  | 未対応 API / ビルド非含有 | 採用ビルドに無いモジュール/関数を呼ぶ | デモ単位で「この機能は未対応」と表示し、他デモへ誘導。事前にビルドの提供関数を確認 |
  | 処理例外 | OpenCV 内部エラー、不正パラメータ組合せ | そのデモのみエラー表示。パラメータを既定へ戻す導線 |
  | 入力不正 | 非画像ファイル、巨大すぎる/破損画像、0px | アップロード時に検証し、明確なメッセージで拒否 |
  | メモリ不足 | 大画像 + 重処理でクラッシュ近接 | 上限縮小を案内、解像度低減を促す (第 8/9 節) |
  | ブラウザ非対応 | WASM/必要 API 非対応 | サポート対象を案内 (第 13 節) |

- **例外捕捉の方針**:
  - OpenCV.js は例外を数値の例外ポインタとして投げることがあり、JS の `catch` で得られる値が分かりにくい場合がある。捕捉時はエラー内容を可能な範囲で人間可読化し、開発ログに詳細を残す。**要調査** (採用ビルドの例外形式)。
  - 例外発生時もメモリ解放を行う (第 6 節の try/finally / スコープラッパ)。
- **UI の堅牢性**:
  - React の Error Boundary で、デモ単位のクラッシュがアプリ全体を落とさないようにする。
  - 処理失敗時も入力画像と他の操作は保持し、復帰可能にする。
- **フィードバック**: ローディング / 成功 / 失敗 / 警告の状態を一貫した UI で提示し、`aria-live` でも通知する (第 10 節)。

## 13. ブラウザ互換性

### 要件

- WebAssembly に対応した主要モダンブラウザで動作すること。サポート対象を明示する。
- カメラ機能 (`getUserMedia`) は HTTPS が必須である点を満たす。

### 方針 / 実装指針

- **サポート対象 (目安)**:

  | ブラウザ | サポート | 注意点 |
  | --- | --- | --- |
  | Chrome (デスクトップ/Android) | 対応 | 基準環境 |
  | Edge (Chromium) | 対応 | Chrome に準拠 |
  | Firefox | 対応 | WASM 対応。SharedArrayBuffer 利用時は COOP/COEP 必須 |
  | Safari (macOS) | 対応 | WASM 対応。メモリ上限・一部 API 差に注意 |
  | iOS Safari (および iOS の全ブラウザ) | 対応 (制約あり) | メモリ制約が厳しくクラッシュしやすい。`getUserMedia` は iOS のバージョン/設定に依存。WebKit ベース固定 |
  | 旧 IE | 非対応 | WASM 非対応 |

- **WASM 対応**: 起動時に WebAssembly 対応を検出し、非対応時は明確に「未対応ブラウザ」案内を出す (処理を試みない)。
- **getUserMedia / HTTPS**: カメラ入力は secure context (HTTPS、または localhost) でのみ動作する。本番は HTTPS 配信を必須とする (静的ホスティング各社は既定で HTTPS)。
- **Safari / iOS の注意**:
  - メモリ制約が厳しいため、上限解像度を保守的に (第 8/9 節)。
  - `SharedArrayBuffer` / WASM threads のサポート状況、COOP/COEP 要件は Safari で特に差が出やすく **要調査**。MVP は単一スレッド前提で iOS Safari を含める。
  - 一部の Canvas/ImageBitmap オプションや色管理の差異に留意し、実機確認を行う。
- **明示**: サポート対象ブラウザと既知の制約を、サイト内 (FAQ/About) に明記する。

## 15. 品質・テスト方針 (概要)

### 要件

- 各デモの処理が、想定パラメータ範囲で破綻なく動作し、データ整合性を保つことを検証する。主要な見た目の崩れに気付ける仕組みを持つ。

### 方針 / 実装指針

- **ユニットテスト**:
  - パラメータスキーマ (型・範囲・既定値) の妥当性、境界値での処理関数の安定動作 (例外を投げない/出力サイズ・型が想定どおり) を検証する。
  - メモリ解放 (実行前後でリークしない) の検証を目標とする (検出手段は第 6 節、確立は要調査)。
  - OpenCV.js を必要とするテストは、ヘッドレス環境での WASM 読み込みが前提となる。実行環境構成は **要調査**。
- **ビジュアル回帰 (可能性)**:
  - 代表入力 + 既定パラメータでの出力画像をスナップショットとして保持し、変更時の差分を検出するビジュアル回帰テストを検討する。OpenCV/WASM のバージョン差で微小なピクセル差が出るため、完全一致でなく許容誤差 (しきい値) ベースの比較とする。MVP では主要デモに限定し、段階的に拡大する。
- **手動確認**:
  - 主要ブラウザ・実機 (特に iOS Safari、Android Chrome) での手動チェックリストを用意し、リリース前に確認する (ロード、処理、メモリ、レイアウト、アクセシビリティの主要観点)。
- 本節は概要であり、詳細なテスト計画は別途定める。

## 16. 国際化 (i18n) 方針 (概要)

### 要件

- MVP は日本語 UI を前提とする。ただし将来的な英語化・多言語化を阻害しない構造とする。

### 方針 / 実装指針

- **日本語前提 + 英語名併記**:
  - UI テキストは日本語を基本とするが、各デモ/カテゴリ/OpenCV 機能名は英語名 (関数名や一般的な英語呼称) を併記する。これは技術用語の検索性・正確性のため (例: 「ガウシアンぼかし (GaussianBlur)」)。
- **将来 i18n を見据えた構造**:
  - 文言をコンポーネントへ直接ハードコードせず、メッセージ辞書 (キー → 文言) 経由で参照する設計にしておく。デモ定義のデータ構造に、表示名を `{ ja, en }` のような言語別フィールドで保持する余地を持たせる。
  - 数値・日付の表示やレイアウト方向 (LTR/RTL) は将来課題として認識しておく。MVP では本格的な i18n フレームワーク導入は行わず、構造のみ準備する。
- **`lang` 属性**: ルート HTML に適切な `lang` (MVP は `ja`) を設定する。
- 本節は概要であり、多言語展開を決定した時点で詳細を定める。
