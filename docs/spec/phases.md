# 開発フェーズ定義 (Development Phases)

OpenCV Playground を段階的に構築するためのフェーズ計画です。
各フェーズの `phase` 値は `src/data/opencvDemos.ts` の `phase` フィールドと一致します。

## フェーズ別デモ数（全 91 件）

| フェーズ | テーマ | デモ数 | 主入力 | 主モジュール |
| --- | --- | --- | --- | --- |
| Phase 1 | 静止画1枚の基本処理 | 34 | image | core, imgproc |
| Phase 2 | 輪郭・形状・図形検出 | 16 | image | imgproc |
| Phase 3 | ヒストグラム・幾何変換・実用レシピ | 18 | image | imgproc, photo |
| Phase 4 | Webカメラ・動画処理 | 6 | camera, video | video |
| Phase 5 | 顔検出・物体検出 | 4 | image, camera | objdetect, dnn |
| Phase 6 | 特徴点マッチング・パノラマ | 7 | multiple-images | features2d, calib3d, stitching |
| Phase 7 | DNN・高度な Computer Vision | 6 | image, camera | dnn, ml, calib3d |

> **MVP（44 件）** は Phase 1 の大部分 + Phase 2/3 の厳選機能で構成されます（詳細は `mvp-scope.md`）。
> Phase 1〜3 は静止画 1 枚で完結し、`opencvJsSupport: supported` が中心。Phase 4 以降で入力形態と不確実性が増えます。

## Phase 1: 静止画1枚の基本処理

- **目的**: アプリの土台（アップロード／サンプル選択／Before-After／パラメータ／コード／説明）と、OpenCV の最も基本的な画像操作を完成させる。「画像＝数値配列」を体験させる。
- **含める機能（34 件・代表例）**:
  - 基盤: `image-upload`, `before-after-compare`
  - 基本操作: グレースケール, リサイズ, 回転, 反転, 切り抜き, 明るさ/コントラスト, チャンネル分割, 画像合成, ビット演算
  - 色: RGB/HSV, 色抽出, マスク処理, 色空間ギャラリー
  - フィルタ: 平均/ガウシアン/メディアン/バイラテラル, シャープ化, カスタムカーネル
  - 二値化: 通常/大津/適応的
  - 形態学: 収縮/膨張/オープニング/クロージング/勾配
  - エッジ: Canny, Sobel, Scharr, Laplacian
- **含めない機能**: 輪郭・形状解析（Phase 2）、ヒストグラム/幾何変換（Phase 3）、カメラ/動画（Phase 4 以降）。
- **実装難易度**: ★☆☆（beginner 中心）。ただし**共通基盤の設計品質**がプロジェクト全体を左右するため最重要。
- **ユーザー体験上の価値**: ★★★。これ単体で「OpenCV を触って学べる」プロダクトとして成立。最大のインパクト。
- **技術的リスク**:
  - OpenCV.js（WASM ~8–10MB）の読み込み UX と初期化待ち。
  - `cv.Mat.delete()` のメモリ管理規約を最初に確立しないと後段で破綻。
  - canvas は RGBA、EXIF 向き、奇数カーネルサイズ制約などの「最初のつまずき」。
- **完了条件**:
  - [ ] OpenCV.js のロード・初期化・失敗時表示が動作。
  - [ ] アップロード/サンプル/自動リサイズ/Before-After（並列・スライダ）が動作。
  - [ ] 34 デモがパラメータ調整付きで動作し、メモリリークが無い。
  - [ ] コード/説明/対応バッジ/注意点の共通表示が動作。

## Phase 2: 輪郭・形状・図形検出

- **目的**: 「2値化 → 輪郭検出 → 形状特徴量」というコンピュータビジョンの中核パイプラインを体験させ、計測・カウント・図形認識まで到達する。
- **含める機能（16 件・代表例）**:
  - 輪郭: 輪郭検出, 輪郭描画
  - 形状特徴量: 面積, 周囲長, 多角形近似, 外接矩形, 回転矩形, 最小外接円, 凸包, モーメント・重心, 形状マッチング
  - 図形検出: Hough直線, Hough円, テンプレートマッチング
  - 領域: 連結成分（Connected Components）
  - 入力基盤: Canvas描画入力
- **含めない機能**: ヒストグラム/幾何変換（Phase 3）、Watershed/GrabCut（Phase 3 後半）、動画/検出（Phase 4 以降）。
- **実装難易度**: ★★☆（intermediate）。`MatVector`/`hierarchy` などの解放、結果の重畳描画・数値テーブル化が増える。
- **ユーザー体験上の価値**: ★★★。「物を数える・測る・形を判定する」など"使える"実感が一気に高まる。
- **技術的リスク**:
  - 結果が前段の 2値化品質に強く依存 → 前処理デモとの導線設計が必要。
  - `overlay`/`numeric`/`geometry` 出力の表示コンポーネントの整備。
  - パラメータ感度（Hough・approxPolyDP の epsilon 等）でユーザーが迷いやすい。
- **完了条件**:
  - [ ] 輪郭パイプライン（2値化→findContours→各特徴量）が一連で体験できる。
  - [ ] overlay/numeric/geometry の表示部品が再利用可能な形で完成。
  - [ ] Hough 直線・円、テンプレートマッチングが動作。

## Phase 3: ヒストグラム・幾何変換・実用レシピ

- **目的**: 画像解析（ヒストグラム）と座標変換（アフィン/透視）を加え、「書類スキャン」のような複数機能を組み合わせた実用レシピまで提示する。
- **含める機能（18 件・代表例）**:
  - ヒストグラム: 計算, 均一化, CLAHE, 比較, バックプロジェクション
  - 幾何変換: アフィン, 透視, 平行移動, 書類スキャン風補正
  - エッジ補助: 画像ピラミッド
  - セグメンテーション: Watershed, GrabCut, Flood Fill, k-means減色
  - 写真加工: インペインティング, スタイライズ, セピア, NL-Means除去
- **含めない機能**: カメラ/動画（Phase 4）、検出（Phase 5）、マッチング/パノラマ（Phase 6）。
- **実装難易度**: ★★☆〜★★★。チャート描画（ヒストグラム）、4点ドラッグ UI（透視）、対話的マスク（GrabCut/inpaint）が登場。
- **ユーザー体験上の価値**: ★★☆。「書類スキャン」など実用レシピは強い訴求。一方で一部は専門的。
- **技術的リスク**:
  - チャート描画はアプリ側実装（OpenCV は描画しない）。
  - `photo` モジュール（inpaint/stylization/NL-Means）と `CLAHE`/`calcBackProject` のバインディングが **要調査**（`research-notes.md` 参照）。提供されない場合は代替/非表示の方針が必要。
  - GrabCut/Watershed は重く、対話 UI（ブラシ/4点）の実装コストが高い。
- **完了条件**:
  - [ ] ヒストグラムのチャート表示と均一化前後比較が動作。
  - [ ] 4点ドラッグの透視変換と書類スキャンレシピが動作。
  - [ ] `needs-investigation` 機能は実機検証し、未対応なら明示的に「未対応バッジ + 説明」で扱う。

## Phase 4: Webカメラ・動画処理

- **目的**: 入力をライブ映像/動画に拡張し、時間方向の解析（動き）を体験させる。
- **含める機能（6 件）**: Webカメラキャプチャ, 背景差分, フレーム差分, Optical Flow (LK), 密なOptical Flow (Farneback), Meanshift/Camshift。
- **含めない機能**: 検出（Phase 5）、マッチング（Phase 6）、DNN（Phase 7）。
- **実装難易度**: ★★★。`requestAnimationFrame` のフレームループ、毎フレームの Mat 解放、前フレーム保持、性能管理が必須。
- **ユーザー体験上の価値**: ★★★。「自分のカメラが動く」インパクトは大きい。
- **技術的リスク**:
  - getUserMedia は **HTTPS 必須**・権限許可が必要。
  - 毎フレーム処理での**メモリリーク**と GC 圧。端末性能差で frame rate が不安定。
  - Farneback/CamShift の opencv.js バインディングが `likely-supported`（要確認）。
  - 重い処理は **Web Worker 化**が事実上必須になる可能性。
- **完了条件**:
  - [ ] カメラ開始/停止/権限エラー処理が動作。
  - [ ] フレームループで Mat が毎フレーム確実に解放される（長時間でもリークしない）。
  - [ ] 背景差分・LK フローが実用 frame rate で動作。

## Phase 5: 顔検出・物体検出

- **目的**: 学習済みカスケード/DNN を使った「検出」を体験させる。モデル/カスケードのロードという新しい概念を導入。
- **含める機能（4 件）**: 顔検出（Haar）, 目検出（Haar）, QRコード検出, DNN顔検出。
- **含めない機能**: 特徴点マッチング（Phase 6）、汎用 DNN 分類/検出（Phase 7）。
- **実装難易度**: ★★★。Emscripten 仮想 FS へのカスケード XML / モデルのロード、検出枠の重畳、カメラ連携。
- **ユーザー体験上の価値**: ★★★。顔検出は最も"AI らしい"体験で訴求力が高い。
- **技術的リスク**:
  - カスケード XML / モデルファイルの取得（CORS・サイズ）と FS 配置。
  - `QRCodeDetector` / DNN 顔検出（YuNet 等）は `needs-investigation`。
  - DNN はモデルサイズ・推論速度・前処理（blobFromImage の正規化）の壁。
- **完了条件**:
  - [ ] カスケード XML のロードと顔/目検出が画像・カメラで動作。
  - [ ] モデル/カスケードのロード進捗・失敗表示が動作。
  - [ ] QR・DNN 顔検出は実機検証し、可否を明示。

---

## Phase 6: 特徴点マッチング・パノラマ

- **目的**: 複数画像入力を導入し、特徴点による画像間の対応付け・位置合わせ・パノラマ合成を体験させる。
- **含める機能（7 件）**: Harrisコーナー, Shi-Tomasiコーナー, ORB特徴点, 特徴量マッチング, Homography推定, SIFT特徴点, パノラマ合成。
- **含めない機能**: DNN/ML（Phase 7）、キャリブレーション（Phase 7）。
- **実装難易度**: ★★★。複数画像 UI、`KeyPointVector`/`DMatchVector` 等の管理、`drawMatches` の表示。
- **ユーザー体験上の価値**: ★★☆。対応点を線で結ぶ可視化は理解効果が高いが、やや専門的。
- **技術的リスク**:
  - `SIFT` の opencv.js 含有が `needs-investigation`（ORB を既定にする）。
  - `Stitcher`（パノラマ）はデフォルトビルド非含有の可能性が高く `needs-investigation` → 無ければ ORB+findHomography で自前合成。
  - マッチングの誤対応（RANSAC での除去が前提）。
- **完了条件**:
  - [ ] ORB + BFMatcher で 2 枚画像の対応点可視化が動作。
  - [ ] findHomography による位置合わせが動作。
  - [ ] パノラマは Stitcher 可否を検証し、不可なら自前実装か明示的に保留。

## Phase 7: DNN・高度な Computer Vision

- **目的**: 深層学習推論（分類/検出）と古典機械学習、カメラキャリブレーションなど、最も高度で重い領域を扱う。
- **含める機能（6 件）**: DNN画像分類, DNN物体検出, k-meansクラスタリング, kNN手書き数字認識, チェスボードコーナー検出, カメラキャリブレーション。
- **含めない機能**: （最終フェーズ。以降は個別機能の拡充）。
- **実装難易度**: ★★★★。モデル取得・前処理・出力デコード・NMS、複数画像キャリブレーション、学習データ同梱など総合力が要る。
- **ユーザー体験上の価値**: ★★☆。「ブラウザだけで物体検出」は強い訴求だが、性能・対応の不確実性が高い。
- **技術的リスク**:
  - モデルサイズ（数 MB〜数十 MB）、推論速度、WASM **SIMD/threads**（SharedArrayBuffer + COOP/COEP）の可否が `needs-investigation`。
  - 出力テンソルのデコードがモデル依存で複雑。
  - `ml` モジュール / `calibrateCamera` の opencv.js 提供が `needs-investigation`。
- **完了条件**:
  - [ ] 軽量モデル（MobileNet / YOLO-nano 級）で分類・検出が動作。
  - [ ] モデルのホスティング・キャッシュ・進捗表示が動作。
  - [ ] `needs-investigation` 機能は実機検証し、可否と性能を明示。

## フェーズ運用の方針

- 各フェーズは**前フェーズの基盤を再利用**して積み上げる（特に Phase 1 の共通部品）。
- `needs-investigation` の機能は、該当フェーズ着手時に**実機（出荷する opencv.js）で検証**し、結果を `opencvJsSupport` に反映する。
- フェーズ完了ごとにメモリリーク検査（長時間操作後のヒープ）を実施する。
- Phase 4 着手時に **Web Worker 化**の方針を確定する（`non-functional-requirements.md` 参照）。
